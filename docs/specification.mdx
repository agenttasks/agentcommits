## File location

A project declares its commit conventions by placing a `COMMIT.md` file in one of these locations, checked in order:

1. `.agents/commits/COMMIT.md`
2. `COMMIT.md` (project root)

Agents MUST check these locations in order and use the first file found. If no `COMMIT.md` exists, agents SHOULD fall back to [Conventional Commits 1.0.0](https://www.conventionalcommits.org/en/v1.0.0/) defaults.

## `COMMIT.md` format

The `COMMIT.md` file MUST contain YAML frontmatter followed by Markdown content.

### Frontmatter

| Field | Required | Constraints |
|-------|----------|-------------|
| `name` | Yes | Max 64 characters. The project or repository name. |
| `spec` | Yes | The spec version. Must be `agent-commits/1.0`. |
| `types` | No | List of allowed commit types. Defaults to the [Conventional Commits set](#default-types). |
| `scopes` | No | List of allowed scopes. If omitted, any scope is allowed. |
| `scope-required` | No | Boolean. If `true`, commits MUST include a scope. Defaults to `false`. |
| `attribution` | No | Agent attribution requirements. See [attribution](#attribution-field). |
| `breaking-changes` | No | Breaking change policy. See [breaking changes](#breaking-changes-field). |
| `description-max-length` | No | Maximum length for the subject line. Defaults to `72`. |
| `metadata` | No | Arbitrary key-value mapping for additional metadata. |

<Card>
**Minimal example:**

```markdown COMMIT.md
---
name: my-project
spec: agent-commits/1.0
---
```

**Example with all fields:**

```markdown COMMIT.md
---
name: my-project
spec: agent-commits/1.0
types: [feat, fix, docs, chore, refactor, test, ci, build, perf, style, revert]
scopes: [api, cli, docs, config]
scope-required: false
description-max-length: 72
attribution:
  require-agent-footer: true
  require-co-authored-by: false
  require-session: false
breaking-changes:
  require-footer: true
metadata:
  maintainer: example-org
  changelog-tool: release-please
---
```
</Card>

#### `name` field

The required `name` field:
- Must be 1–64 characters
- Identifies the project for tooling and validation
- Typically matches the repository name

#### `spec` field

The required `spec` field:
- Must be `agent-commits/1.0` for this version of the specification
- Agents MUST check this field and warn if the version is unrecognized

#### `types` field

The optional `types` field:
- A list of allowed commit type prefixes
- If omitted, defaults to the [full Conventional Commits set](#default-types)
- Agents MUST reject commits with types not in this list

<Card>
**Example:**
```yaml
types: [feat, fix, docs, chore, test]
```
</Card>

##### Default types

When `types` is omitted, the following types are allowed (based on `@commitlint/config-conventional`):

| Type | Description |
|------|-------------|
| `feat` | A new feature (correlates with SemVer MINOR) |
| `fix` | A bug fix (correlates with SemVer PATCH) |
| `docs` | Documentation changes only |
| `style` | Formatting, whitespace, semicolons — no code change |
| `refactor` | Code change that neither fixes a bug nor adds a feature |
| `perf` | Performance improvement |
| `test` | Adding or updating tests |
| `build` | Build system or external dependency changes |
| `ci` | CI/CD configuration changes |
| `chore` | Maintenance tasks that don't modify source or tests |
| `revert` | Reverts a previous commit |

#### `scopes` field

The optional `scopes` field:
- A list of allowed scope values
- If omitted, any scope is allowed (including no scope)
- Agents MUST reject commits with scopes not in this list when the field is present

<Card>
**Example:**
```yaml
scopes: [api, cli, docs, config]
```
</Card>

#### `scope-required` field

The optional `scope-required` field:
- Boolean, defaults to `false`
- When `true`, agents MUST include a scope in every commit message

#### `attribution` field

The optional `attribution` field controls agent attribution requirements:

| Sub-field | Type | Default | Description |
|-----------|------|---------|-------------|
| `require-agent-footer` | Boolean | `false` | Require an `Agent:` footer identifying the agent |
| `require-co-authored-by` | Boolean | `false` | Require a `Co-Authored-By:` header for the agent |
| `require-session` | Boolean | `false` | Require a `Session:` footer linking to the agent session |

<Card>
**Example:**
```yaml
attribution:
  require-agent-footer: true
  require-co-authored-by: true
```

Produces commits like:
```
feat(api): add user endpoint

Implement GET /users/:id with pagination support.

Agent: claude-code
Co-Authored-By: Claude <noreply@anthropic.com>
```
</Card>

#### `breaking-changes` field

The optional `breaking-changes` field:

| Sub-field | Type | Default | Description |
|-----------|------|---------|-------------|
| `require-footer` | Boolean | `false` | Require a `BREAKING CHANGE:` footer (not just `!` suffix) |

When `require-footer` is `true`, agents MUST include a `BREAKING CHANGE:` footer for any breaking change, even if `!` is also used in the type/scope prefix.

#### `description-max-length` field

The optional `description-max-length` field:
- Integer, defaults to `72`
- Maximum character length for the commit subject line (after the `type(scope): ` prefix)
- Agents SHOULD wrap the commit body at this length as well

#### `metadata` field

The optional `metadata` field:
- A map from string keys to string values
- Agents and tooling can use this for additional configuration not defined by the spec
- Use reasonably unique key names to avoid conflicts

### Body content

The Markdown body after the frontmatter contains instructions for agents. There are no format restrictions — write whatever helps agents produce better commits for your project.

Recommended sections:
- Project-specific conventions and style preferences
- Examples of good and bad commit messages
- Common scopes and when to use them
- Guidance on when to split changes into multiple commits

## Commit message format

Agent Commits extends the [Conventional Commits 1.0.0](https://www.conventionalcommits.org/en/v1.0.0/) message format. All rules from that specification apply. This section describes agent-specific extensions.

### Agent attribution footers

Agent Commits defines three optional footers for agent attribution:

```
<type>(<scope>): <description>

<body>

Agent: <agent-identifier>
Co-Authored-By: <agent-name> <agent-email>
Session: <session-url>
```

#### `Agent` footer

Identifies which agent produced the commit:
- Value MUST be a lowercase identifier (e.g., `claude-code`, `cursor`, `github-copilot`, `aider`)
- Projects MAY require this footer via `attribution.require-agent-footer`

#### `Co-Authored-By` footer

Credits the agent as a co-author using the standard git trailer format:
- Value MUST follow `Name <email>` format
- This footer is recognized by GitHub, GitLab, and other platforms for attribution
- Projects MAY require this footer via `attribution.require-co-authored-by`

#### `Session` footer

Links to the agent session for traceability:
- Value SHOULD be a URL pointing to the session where the commit was produced
- Projects MAY require this footer via `attribution.require-session`

### Commit splitting

When an agent makes changes that span multiple logical units, it SHOULD split the work into separate commits. Each commit should represent one coherent change.

Guidelines for agents:
- One bug fix per commit, even if you fixed multiple bugs in a session
- Separate feature additions from refactoring
- Separate formatting/style changes from logic changes
- Group related test additions with the feature or fix they test
- Infrastructure changes (CI, build, config) should be their own commits

## Validation

A valid `COMMIT.md` file:
1. Contains valid YAML frontmatter delimited by `---`
2. Has a `name` field (1–64 characters)
3. Has a `spec` field set to `agent-commits/1.0`
4. Has only recognized fields in the frontmatter (unknown fields trigger a warning)
5. Has valid values for all fields that are present

A valid agent commit message:
1. Follows the [Conventional Commits 1.0.0](https://www.conventionalcommits.org/en/v1.0.0/) specification
2. Uses a type listed in the `types` field (or the default set)
3. Uses a scope listed in the `scopes` field (if the field is present)
4. Includes a scope (if `scope-required` is `true`)
5. Keeps the description under `description-max-length` characters
6. Includes required attribution footers (if configured)
7. Includes a `BREAKING CHANGE:` footer for breaking changes (if `breaking-changes.require-footer` is `true`)
